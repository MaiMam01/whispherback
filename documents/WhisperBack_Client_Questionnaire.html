Section A · Scope & Limits
How to answer: Write your reply in the “Your answer” box under each question, or reply inline by email — whichever is easier for you. Short answers are fine. Where a recommended default is shown in the hint, a simple “Agreed” is enough.

A · Branding & Platforms

Foundational decisions that affect the UI, stores, and framework choice.

A1
Which brand spelling is correct — WhisperBack or WhispherBack?
Both spellings appear in the document. We need one final spelling to use across the app, stores, and domain.
A2
Can you share the final logo, brand colour (hex codes), and typography?
The document references a colour swatch image. We need the exact hex values and any font files or licences.
A3
Target platforms — Android, iOS, or both? Any minimum OS version?
Platform choice affects the framework (native vs Flutter vs React Native) and how background audio is handled.
A4
Primary target audience and launch regions?
This drives localisation, right-to-left language support, and default prayer-time settings.

B · Open Limits from the Spec

The current document left a few numeric limits open. Please confirm.

B1
Maximum number of playlists for the Premium plan? (Basic is 20)
We suggest 100 for Premium. Please confirm, or set your own number.
B2
Maximum clips per playlist? (Document says 20–50, “or more?”)
Affects UI pagination and memory use. Recommended: 50.
B3
Maximum scheduled playlists active at the same time? (Document says 10–20)
A higher limit increases risk of time conflicts and battery use. Recommended: 10.

Section B · Audio · Scheduling · Sleep Mode

C · Accounts & Monetisation

How users sign in and how Basic vs Premium is sold.

C1
Is login optional or required to use the app?
Google login is mentioned. Can users use the app fully offline without creating an account?
C2
What is the monetisation model — free, one-time purchase, or subscription?
This affects in-app purchase integration, receipt validation, and cloud-storage budgeting.
C3
Which features are Premium-only, beyond cloud sync?
For example: higher playlist count, Prayer Mode, web access, higher recording quality.

D · Audio & Recording

The document says recording quality “should be good” — we need concrete numbers.

D1
What recording quality should the in-app recorder produce?
Recommended: AAC, 44.1 kHz, 128 kbps, mono. Please confirm or request higher.
D2
Maximum clip length and file size?
For example, 5 minutes or 10 MB per clip. Needed for storage and upload limits.
D3
Which import formats should be supported? (MP3, M4A, WAV, OGG…)
Also: should we support DRM-protected files? (We recommend: no.)

E · Scheduling & Sleep Mode

Background playback and Prayer Mode have important trade-offs.

E1
If a scheduled clip’s time arrives during a phone call or other playback, should we skip it, queue it, or delay it?
Affects audio-focus logic and user expectation. Recommended: skip with a small notification.
E2
On iOS, exact-time background playback is restricted. Is a ±1–2 minute drift acceptable?
Strict exact timing on iOS requires the user to tap a notification — this changes the UX.
E3
Which prayer-time calculation method should be the default? (MWL, ISNA, Umm al-Qura, Karachi, Egyptian…)
Users should also be able to change it in settings, along with the madhab for Asr.
E4
What should happen if the internet is unavailable at prayer time?
Recommended: bundle offline prayer-time tables so Prayer Mode still works. Please confirm.

Section C · Cloud · UX · Delivery

F · Cloud Sync & Website

Premium users can upload playlists to the cloud and share via a website.

F1
On the website, should visitors download playlists, only stream them, or both?
This affects storage costs, DRM needs, and privacy rules.
F2
Are shared playlists public by default, private by default, or user-configurable?
The spec says “for others to download or play” — please confirm the visibility model.
F3
Preferred cloud provider or region? Any data residency requirements such as GDPR?
We recommend AWS or Firebase. Needed for cost planning and compliance.

G · UX, Accessibility & Localisation

Icon-only UI needs care for accessibility and non-English markets.

G1
Which languages should version 1 ship in? Is right-to-left (Arabic / Urdu) needed?
Prayer Mode strongly suggests RTL markets — please confirm.
G2
The UI is icon-only. Shall we add accessibility labels and an optional “show labels” mode?
Strongly recommended — it also helps pass app-store reviews.
G3
Should the app support dark mode, or only the current purple theme?
Low cost if designed in from the start; expensive to retrofit later.

H · Delivery & Scope

Timeline, accounts, and the proposed v1 scope.

H1
What is your target launch date and any key milestones such as beta or store submission?
We will align the sprint plan and the MVP scope with your dates.
H2
Proposed v1.0 scope: Active / Inactive · Playlists · Manual playback · Scheduling · Shuffle · basic Sleep Mode — local only. Is this acceptable, or do you want Prayer Mode and Cloud Sync in v1 as well?
Phasing keeps v1 lean and lets us prove background-audio reliability before adding complexity.
H3
Any additional features you have in mind that are not in the current document?
Much better to surface new ideas now than during QA.
